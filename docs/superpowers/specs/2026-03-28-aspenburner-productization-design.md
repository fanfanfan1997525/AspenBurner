# AspenBurner 产品化设计

> 设计方法：基于 superpowers 的方案化设计流程收敛，结合仓库审计结论与后续多模型审议。

版本: v0.5.0  
日期: 2026-03-28

## 1. 目标

把 AspenBurner 从脚本工具重构为一个可直接运行的 Windows 专用软件，解决以下核心问题：

1. 让用户明确知道软件是否在运行、是否已生效、当前数据是否可信。
2. 保持现有准心与 CPU 角标能力，并减少 PowerShell 宿主带来的稳定性和维护问题。
3. 在不破坏现有配置和入口习惯的前提下，提供统一的应用生命周期、设置界面和运行状态。

## 2. 非目标

- 本轮不做安装器、自动更新器、云同步或账号系统。
- 本轮不做多游戏模板库。
- 本轮不做主题系统和复杂视觉皮肤。
- 本轮不做内核级注入、驱动或绕过游戏保护的能力。

## 3. 方案比较

### 方案 A：继续 PowerShell 产品化

- 优点：迁移最快。
- 缺点：应用生命周期、单实例、托盘、IPC、崩溃恢复、分发和长期维护都不理想。
- 结论：不选。

### 方案 B：迁移到 C# WinForms

- 优点：对当前需求最匹配，透明 overlay、托盘、Win32 互操作、动态加载 `Control Center` 托管程序集都直接可做。
- 缺点：需要新工程和 SDK。
- 结论：主推方案。

### 方案 C：迁移到更原生的 Win32/C++/Rust

- 优点：理论上更轻。
- 缺点：重写成本过高，当前收益不足。
- 结论：当前阶段不选。

## 4. 最终决策

采用 `C# WinForms` 构建单实例桌面软件，并保留现有脚本入口作为兼容壳层。

### 决策理由

- 这是当前最低风险的“专门软件化”路径。
- 可以用更稳的应用模型解决真正的问题，而不是继续在脚本层缝补。
- 与当前配置、目标能力和用户习惯兼容性最好。

## 5. 系统架构

### 5.1 进程模型

软件收敛为一个主进程，并通过本地命名管道接收次实例命令：

- 主进程：`AspenBurner.exe`
  - 单实例互斥
  - 托盘图标与菜单
  - 透明准心窗体
  - 透明状态角标窗体
  - 设置窗体
  - 命令服务器（处理次实例带来的 `show-settings` / `stop` / `preview` / `health` 命令）

提权策略明确如下：

- `AspenBurner.exe` 默认 manifest 为 `asInvoker`
- 兼容入口 `Start-Crosshair.ps1/.cmd`、`Configure-Crosshair.ps1/.cmd`、`Stop-Crosshair.ps1/.cmd` 继续作为正式公开接口保留
- 为保持当前用户习惯和游戏内可见性，兼容入口可以按现有路径触发 UAC 并拉起 elevated 主实例
- 次实例与主实例通信统一走命名管道；如果兼容入口发现需要与 elevated 主实例通信，则自身也走提权链，避免低完整性实例与高完整性主实例之间的兼容歧义

兼容入口：

- `Start-Crosshair.cmd` -> 启动主程序
- `Configure-Crosshair.cmd` -> 请求主程序打开设置窗口
- `Stop-Crosshair.cmd` -> 请求主程序退出

### 5.2 模块边界

#### 应用层

- `Program`
- `AspenBurnerApplicationContext`
- `AppCommandServer`
- `AppCommandClient`

职责：

- 互斥、托盘、命令路由、窗体生命周期、退出清理。

#### 配置层

- `CrosshairConfig`
- `CrosshairConfigService`
- `CrosshairConfigValidator`
- `CrosshairConfigMigrator`

职责：

- 读取/校验/保存 `config/crosshair.json`
- 保持向后兼容
- 原子写入
- 保证设置 UI 不会把合法旧配置静默夹断；尤其保留现有 `OffsetX/OffsetY = -4000..4000` 支持域
- 配置引入 `ConfigVersion`，在读取时执行迁移，而不是把旧文件当成“恰好还能读”的幸运结果

#### 领域层

- `CrosshairGeometry`
- `StatusOverlayPlacement`
- `StatusTextFormatter`

职责：

- 纯逻辑计算，可单元测试。

#### 运行时层

- `OverlayRuntime`
- `TargetWindowMonitor`
- `IForegroundWindowSource`
- `ITickSource`

职责：

- 前台窗口检测
- overlay 显示/隐藏
- 防抖与位置更新
- 为 TDD 提供可替换 seam，不把 Win32 调用和时钟源写死在逻辑层

#### 遥测层

- `ControlCenterCpuProvider`
- `FallbackCpuProvider`
- `CpuStatusService`

职责：

- 优先读取 `Control Center`
- 失败时回退通用频率方案
- 暴露数据源状态与最后更新时间
- 将厂商链路封装在 provider 边界内，主循环只消费快照，避免将第三方程序集细节扩散到 UI 和 overlay 线程

#### UI 层

- `CrosshairOverlayForm`
- `StatusOverlayForm`
- `SettingsForm`
- `PreviewCanvas`

职责：

- 真实 overlay 绘制
- 设置编辑、实时预览与运行状态反馈

#### 诊断层

- `AppLogger`
- `HealthSnapshot`

职责：

- 统一文件日志
- 记录启动、命令、配置迁移、provider 初始化、provider 回退、预览开始/结束、退出原因
- 为设置页和兼容脚本提供可查询健康状态

## 6. 核心交互逻辑

### 6.1 首次启动

1. 启动应用。
2. 若配置不存在，则创建默认配置。
3. 托盘图标出现。
4. 设置窗口默认打开。
5. 页面顶部显示运行状态：
   - `运行中 / 已暂停 / 已停止`
   - `检测到目标窗口 / 未检测到目标窗口 / 正在桌面预览`
   - `CPU 数据源：Control Center / Fallback / 不可用`
   - `数据状态：Fresh / Stale / Unavailable`

### 6.2 常规运行

1. 主程序常驻托盘。
2. 仅当目标游戏处于前台时显示准心和角标。
3. 用户可通过托盘打开设置、临时桌面预览、暂停显示、退出。
4. 托盘 tooltip 与菜单标题必须明确区分四种状态：
   - `已运行，等待目标窗口`
   - `已运行，目标窗口已命中`
   - `已暂停`
   - `遥测异常`

### 6.3 设置修改

1. 用户拖动滑块或输入数值。
2. 预览区立即变化。
3. 真实运行中的 overlay 同步更新，不再要求手工重启。
4. 配置原子保存到 `crosshair.json`。
5. 若输入超出 UI 展示范围，不允许静默夹断合法历史值；必须显式提示并给出恢复路径。
6. 设置页必须给出明确回执：
   - `已保存并应用`
   - `已保存，等待目标窗口可见`
   - `保存失败`
   - `输入被拒绝/被校正`

### 6.4 桌面预览

新增“临时桌面预览 8 秒”能力：

- 即使当前不在游戏前台，也可强制显示一次真实 overlay。
- 用于确认准心大小、颜色、角标位置和透明度。
- 这是为解决“没看到到底是没启动还是没命中目标窗口”而加的诊断交互。
- 预览期间必须有可见状态提示：`桌面预览中（剩余 Ns）`。
- 预览结束后必须给出结束原因：`预览结束，已恢复目标窗口检测模式`。

### 6.5 数据可信度反馈

设置页明确展示：

- 当前 CPU 频率
- 当前 CPU 温度
- 数据源名称
- 最近更新时间
- 当回退或不可用时的可读提示
- 数据新鲜度标签：
  - `Fresh`: 最近一次成功读取在 3 秒内
  - `Stale`: 超过 3 秒未刷新，但保留最近一次值
  - `Unavailable`: 当前没有可用值，不显示旧值冒充真值

避免再次出现“`TZ 35C` 看起来像真值”的交互误导。

## 7. 稳定性设计

### 7.1 单实例与命令转发

- 用命名互斥锁保证只有一个主实例。
- 次实例不直接拉第二套 overlay，而是把命令发送给主实例后退出。

### 7.2 原子保存

- 配置先写入临时文件，再替换正式文件。
- 避免频繁保存时产生损坏 JSON。
- 实时应用通过进程内状态和命名管道命令完成，不再依赖“写文件 + 轮询重读”作为主同步通道。

### 7.3 低频遥测

- CPU 遥测默认 `1000-1500ms` 刷新。
- 不在高频 UI 刷新里做重查询。
- 减少对游戏帧时间的干扰。

### 7.4 错误隔离

- `Control Center` 数据源初始化失败不能拖垮整个应用。
- 遥测失败只影响角标内容，不影响准心显示。
- 所有初始化失败都要落日志并在设置页暴露可见状态，不能继续使用静默空 `catch`。

### 7.5 兼容保底

- 现有脚本入口继续保留。
- 现有 JSON 字段继续兼容。
- 如果新程序不可用，至少不破坏用户现有配置文件。

## 8. 用户体验设计

### 8.1 必须解决的体验问题

- 用户能知道程序是不是活着。
- 用户能知道设置是否真的打到了真实 overlay。
- 用户能知道 CPU 数据是否可信。
- 用户能在不进游戏时做真实效果诊断。

### 8.2 UI 结构

设置窗口采用三段布局：

- 左侧：真实效果预览
- 右侧上半：准心设置
- 右侧下半：CPU 状态与运行状态

底部按钮：

- `恢复默认`
- `保存配置`
- `桌面预览 8 秒`
- `退出程序`

### 8.3 运行态可见性

托盘菜单至少包含：

- `显示设置`
- `桌面预览 8 秒`
- `暂停显示`
- `退出`

语义必须固定：

- 关闭设置窗口：只关闭窗口，不退出程序，回到托盘
- `暂停显示`：停止显示准心和角标，但保留主进程、托盘和遥测状态
- `Stop-Crosshair` / `stop` 命令：请求主程序优雅退出
- `退出程序`：与 `stop` 等价，都是彻底结束主程序

## 9. 向后兼容

保持以下不变：

- `config/crosshair.json`
- `Start-Crosshair.cmd`
- `Configure-Crosshair.cmd`
- `Stop-Crosshair.cmd`
- `Start-Crosshair.ps1`
- `Configure-Crosshair.ps1`
- `Stop-Crosshair.ps1`

变化点：

- 它们不再直接操控 PowerShell overlay 进程，而是转为控制 `AspenBurner.exe`。

## 10. 测试策略

### 10.1 单元测试

覆盖纯逻辑：

- 配置默认值与校验
- 颜色解析
- 准心几何
- 状态角标锚点与拖拽换算
- CPU 状态文本格式化
- 命令解析

### 10.2 集成验证

- 单实例与命令转发
- 配置保存后真实 runtime 更新
- 桌面预览命令可触发真实 overlay
- 旧脚本入口可用
- 兼容脚本在提权/取消提权时能给出稳定退出语义
- 日志与健康状态可反映 provider 初始化、回退和失败

### 10.3 手工验证

- 托盘行为
- 设置窗口实时生效
- 三角洲前台显示
- 非目标窗口隐藏
- `Control Center` 可用时显示真实温度

## 11. 风险与缓解

### 风险 1：.NET 环境缺失

- 缓解：把 `.NET SDK` 安装与验证前置为迁移底座任务，不把它夹在功能开发里顺手完成。

### 风险 2：`Control Center` 反射链路与当前脚本行为不完全一致

- 缓解：先做独立 provider 层与验证基线，失败时回退，不把它耦进 overlay 主循环。

### 风险 3：UI 实现过重导致交付拖慢

- 缓解：本轮只做必要 UI，不做复杂主题和动画。

### 风险 4：旧入口被破坏

- 缓解：保留脚本，改为兼容壳层。

## 12. 迁移策略

本轮不采用“一步到位重写”，而采用四阶段迁移：

### 阶段 A：补底座

- 安装并验证 `.NET SDK`
- 建立新工程
- 明确现有入口、配置和用户语义的兼容契约

### 阶段 B：补兼容护栏

- 用当前 PowerShell 逻辑生成兼容基线
- 把配置、几何、文本、入口语义写成可自动验证的契约测试

### 阶段 C：迁移可控部分

- 先迁移纯逻辑、单实例、托盘、设置、桌面预览和 overlay 显示
- 让软件先具备“可运行、可观察、可配置”的产品闭环

### 阶段 D：迁移高风险部分

- 最后迁移 `Control Center` 遥测 provider
- 以“真机 smoke + 明确回退状态”为验收，不把高风险链路作为迁移起点

## 13. 验收标准

满足以下即视为本轮设计落地成功：

1. 本地存在可运行的 `AspenBurner.exe`。
2. 软件单实例常驻，带托盘。
3. 设置窗口可实时修改真实 overlay，而不是只改预览。
4. 支持桌面预览，不必进游戏才能判断是否生效。
5. CPU 状态能显示真实 `Control Center` 数据，失败时明确提示回退。
6. 运行/暂停/退出/等待目标窗口/桌面预览这几种状态都能被用户明确辨识。
7. 旧 `cmd` 入口仍可用。
8. 自动测试通过，且最终有一次真实用户路径模拟验证通过。

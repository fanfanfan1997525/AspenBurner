# AspenBurner 产品化审计

版本: v0.5.0  
日期: 2026-03-28

## 1. 当前形态

当前项目本质上还是一组以 PowerShell 为宿主的脚本工具：

- `CrosshairOverlay.ps1` 负责准心、状态角标、窗口检测、CPU 数据源和消息循环。
- `Configure-Crosshair.ps1` 负责设置面板、预览、自动保存和重启。
- `Start-Crosshair.ps1` / `Stop-Crosshair.ps1` 负责提权与进程拉起/结束。
- `src/CrosshairOverlay.Core.psm1` 提供可测试的几何、配置和格式化逻辑。

它已经能用，但离“专门的软件”还有明显距离。现在最大的问题不是功能缺失，而是：

1. 运行模型仍然像脚本集合，不像一个有状态的桌面应用。
2. 稳定性和可观察性不足，用户经常不知道程序是否真的在运行、是否生效、数据是否可信。
3. 交互链路不完整，很多关键行为依赖脚本重启、UAC 弹窗和外部快捷方式。

## 2. 工程问题

### 2.1 结构耦合过重

- `CrosshairOverlay.ps1` 已到 `1160` 行，集中了配置加载、原生互操作、CPU 遥测、前台窗口检测、窗体绘制和主循环。
- `Configure-Crosshair.ps1` 已到 `933` 行，配置编辑、预览绘制、自动保存、拖拽定位和按钮动作全部耦在一个脚本里。
- 结果是：代码可测部分只覆盖了一小段纯函数逻辑，真正高风险的窗体生命周期、进程控制和错误恢复没有被隔离。

### 2.2 运行时没有真正的应用宿主

- 现在没有单实例应用上下文、托盘、命令路由或统一生命周期管理。
- 启动/设置/停止分别依赖不同脚本入口，不共享运行状态。
- “设置面板改完后是否真的打到运行中的准心”这个问题，本质上是因为项目没有稳定的主进程和进程内状态总线。

### 2.3 进程控制方式脆弱

- `Start-Crosshair.ps1` 和 `Stop-Crosshair.ps1` 通过扫描 PowerShell 命令行并 `Stop-Process` 的方式管理进程。
- 这种方式依赖命令行匹配，容错差、可观测性差，也不适合软件化后的多命令入口。

### 2.4 缺少结构化诊断

- 目前没有统一日志目录、日志级别、崩溃落盘、配置错误快照、遥测数据源状态记录。
- 一旦用户说“看不到”“闪”“不稳定”，只能靠手工排查，工程效率低。

### 2.5 缺少配置版本化与迁移

- `config/crosshair.json` 直接靠默认值 merge 和字段校验工作。
- 一旦后续新增字段、重命名字段或引入更复杂的目标进程规则，现有 schema 没有迁移层。

### 2.6 测试覆盖面偏窄

- 当前 `46/46` 的 Pester 测试主要覆盖 `src/CrosshairOverlay.Core.psm1` 中的纯逻辑。
- 高风险行为没有自动验证：单实例、命令转发、设置热应用、窗体显示/隐藏、配置原子保存、托盘行为、启动脚本兼容性。

## 3. 稳定性问题

### 3.1 PowerShell + WinForms 定时轮询不是理想宿主

- 现在主循环靠 `System.Windows.Forms.Timer` 在 PowerShell 宿主里跑。
- 功能上能成立，但对长期驻留、统一异常恢复、资源管理和产品化分发都不友好。

### 3.2 UAC 行为割裂

- 游戏需要管理员层级时，启动/停止链路通过脚本提权解决。
- 但用户感知上仍然是“为什么有时要点 UAC、有时看不到、有时停不掉”。
- 这不是单个 bug，而是宿主模型不统一导致的体验不确定性。

### 3.3 遥测数据源可信度缺少显式状态

- 现在虽然已优先接 `Control Center` 真温度链路，但用户界面上没有稳定呈现“当前数据源是谁、是否已回退、最后更新时间”。
- 数据一旦波动，用户无法判断是正常瞬时波动，还是数据源退化。

### 3.4 配置写入不是原子切换

- `Configure-Crosshair.ps1` 直接 `Set-Content` 写 JSON。
- 在频繁保存、异常中断或多入口同时改配置的情况下，存在写半截或状态竞争的风险。

### 3.5 前台窗口命名策略偏硬编码

- 目标进程名当前在脚本里硬编码为少量值。
- 这对于单一游戏够用，但软件化后应下沉为可配置目标列表，并提供运行态可见性。

### 3.6 旧脚本入口仍可能造成状态分裂

- 现在 Start/Stop/Configure 都是外部控制器。
- 如果项目继续在这一模型上叠功能，状态一致性会越来越差。

## 4. 用户体验与交互问题

### 4.1 用户不知道程序是否在运行

- 当前桌面默认不显示准心，设置面板预览也不是实际运行效果。
- 用户很容易把“没看到”理解成“没启动”，或者把“预览变了”理解成“真实生效了”。

### 4.2 用户不知道配置是否真正应用

- 历史上已经发生过“预览变了，真实没变”的问题。
- 即使现在加了热重载，界面仍缺少“已连接到运行实例/已应用到实时 overlay”的明确反馈。

### 4.3 用户不知道 CPU 数据是否可信

- 温度、频率是用户高度敏感的信息，但当前 UI 上没有数据源状态、更新时间、回退提示。

### 4.4 关键操作分散在多个脚本和快捷方式

- 启动、停止、设置、查看状态不是一个闭环，而是多个入口拼起来。
- 这会导致“桌面上点了某个快捷方式，但系统到底发生了什么”不够明确。

### 4.5 缺少托盘与最小可见反馈

- 常驻类软件通常至少要有托盘状态、菜单和运行提示。
- 现在没有统一的应用存在感。

### 4.6 缺少桌面预览/诊断模式

- 对准心类产品来说，“给我看一下现在的真实效果”是高频需求。
- 现在只能依赖游戏前台，诊断成本高。

## 5. 路线比较

### 5.1 路线 A：继续用 PowerShell 产品化

优点：

- 迁移成本最低。
- 可复用现有逻辑最多。

缺点：

- 宿主、生命周期、分发、单实例、日志、托盘和 IPC 都不自然。
- 长期维护成本高。
- “像脚本集合”这一根问题不会消失。

结论：不推荐作为主路线。

### 5.2 路线 B：迁移到 C# WinForms 专用桌面应用

优点：

- 与当前透明窗体、托盘、Win32 互操作、`Control Center` 托管程序集反射最兼容。
- 运行时比 PowerShell 宿主更轻、更稳定、更适合常驻。
- 可以在不破坏现有 JSON 配置和脚本入口的前提下平滑迁移。

缺点：

- 需要引入 .NET SDK 与新工程。
- 需要把现有核心逻辑重新组织成可维护的类型。

结论：推荐。它是“收益/风险比”最合理的产品化路径。

### 5.3 路线 C：更原生的 Win32/C++/Rust 方案

优点：

- 理论上占用更低、控制力更强。

缺点：

- 重写成本和验证成本过高。
- 当前项目的真实瓶颈不是语言性能，而是应用模型与工程质量。

结论：当前阶段不值得。

## 6. 审计结论

### 核心判断

值得做，而且应当直接进入“专用桌面软件”重构。  
原因不是为了追求“看起来更像软件”，而是当前问题的根已经从功能问题变成了宿主模型问题。

### 推荐方向

采用 `C# WinForms + 单实例应用上下文 + 托盘 + 原生透明 overlay + 统一设置窗口 + JSON 配置兼容层`。

### 必须保留的兼容点

- 保留 `config/crosshair.json` 作为主配置文件。
- 保留 `Start-Crosshair.cmd` / `Configure-Crosshair.cmd` / `Stop-Crosshair.cmd` 作为兼容入口。
- 保留当前颜色、长度、间距、粗细、四臂开关、状态角标、`Control Center` 遥测优先级等已存在能力。

### 第一阶段不做的事

- 不先做安装包和自动更新。
- 不先扩展成多游戏配置中心。
- 不做重型 UI 框架迁移。
- 不做和当前目标无关的花哨主题化。
